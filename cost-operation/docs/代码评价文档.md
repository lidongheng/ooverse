# InfraMap 项目代码评价文档

> 评价重点：软件架构设计、价值代码贡献、开发者测试  
> 项目范围：`/Users/donghengli/Desktop/www/infraMap`  
> 技术栈：Vue 3、Vue Router、Pinia/Vuex、Element Plus、ECharts、Vue I18n

## 一、总体评价结论

InfraMap 是一个面向基础设施资源、算力资源、损益分析等场景的数据可视化前端项目。项目整体已经形成了比较清晰的工程分层：入口启动、插件注册、路由守卫、业务视图、通用图表组件、可组合业务逻辑、配置数据和工具函数分别沉淀在独立目录中，能支撑多页面、多指标、多图表的持续演进。

从代码贡献价值看，项目不是简单页面堆叠，而是围绕“地图大屏 + 资源池气泡图 + 指标阈值亮灯 + 路由一致性 + 多环境构建”做了多处工程化抽象。尤其是初始化流程、路由守卫、复杂 ECharts 组件、地图标签和阈值规则处理，体现了对业务复杂度的拆解能力。

从开发者测试看，当前具备构建验证能力，`npm run build` 可以成功产出 `dist/infraMap`；但自动化测试和 lint 门禁仍不完整，`npm run lint` 当前不可用，项目中也未看到业务单元测试目录。建议后续优先补齐可测试纯函数的单元测试和 CI 质量门禁。

## 二、软件架构设计评价

### 1. 架构分层清晰，职责边界基本明确

项目目录按职责拆分，主结构如下：

```text
src/
├── components/    # 通用组件与业务组件
├── composables/   # 通用组合式能力
├── config/        # 路由、主题、区域坐标等配置
├── locales/       # 国际化资源
├── plugins/       # Element、ECharts、主题、初始化等插件
├── router/        # 路由与守卫
├── stores/        # Pinia 状态
├── utils/         # 通用工具函数
└── views/         # 页面与业务 hooks
```

这种结构的价值是：页面不直接承载所有逻辑，业务规则可以沉淀到 hooks、utils、config 中，后续新增指标、新增页面或调整可视化规则时，不需要大范围改动页面组件。

### 2. 启动流程具备“插件化 + 初始化编排”思路

入口文件将应用创建、插件注册、主题初始化、全局属性注入、初始化接口请求和挂载流程集中在 `bootstrap` 中，避免散落在多个页面中重复处理。

代表代码：

```js
// src/main.js
async function bootstrap() {
  try {
    const app = createApp(App)

    setupElementPlus(app)
    app.use(i18n)
    setupTheme(app)
    app.use(VueDOMPurifyHTML)

    app.config.globalProperties.$echarts = echarts
    app.config.globalProperties.$dayjs = dayjs
    app.config.globalProperties.$cookies = Cookies

    const pinia = createPinia()
    app.use(pinia)

    const initResult = await initializeApp(app)

    if (!initResult.success) {
      throw new Error(initResult.error)
    }

    app.use(store).use(router).mount('#app')
  } catch (error) {
    showInitError(error.message || '应用启动失败')
  }
}
```

评价：  
这段代码体现了清晰的应用生命周期控制。尤其是先注册 Pinia，再执行依赖 store 的初始化请求，能避免状态管理尚未激活就访问 store 的问题。初始化失败统一进入错误展示，也符合用户体验和问题定位要求。

### 3. 初始化模块按业务入口差异做分支编排

初始化逻辑在 `src/plugins/init.js` 中进一步拆解为 URL 信息识别、token 获取、模块判断、接口请求、数据校验等步骤。普通模块会并行请求 Role、Version、ModuleList，特殊模块和 A3 质量地图模块走差异化流程。

项目将 path 与 tab 的映射关系放在 `src/config/routes.js`，再由 `src/router/index.js` 的守卫统一执行校验和重定向。

```js
// src/router/index.js
function beforeEachGuard(to, from, next) {
  if (!guardState.enabled) {
    next()
    return
  }

  const path = to.path
  const tab = to.query.tab
  const otherQuery = { ...to.query }
  delete otherQuery.tab

  const redirectInfo = checkRouteRedirect(path, tab)

  if (redirectInfo) {
    if (!canRedirect()) {
      next()
      return
    }

    guardState.redirectCount++
    guardState.lastRedirectTime = Date.now()

    next({
      path: redirectInfo.path,
      query: { ...redirectInfo.query, ...otherQuery }
    })
    return
  }

  resetRedirectCount()
  next()
}
```

评价：  
这套设计把路由规则从守卫逻辑中抽离出来，新增模块时优先改配置而不是改控制流程；同时通过最大重定向次数和冷却时间降低死循环风险。对多模块入口、外部系统带 tab 参数跳转的场景很有价值。

### 5. 可视化能力沉淀为组件和业务 hooks

项目中 ECharts 不是散落在每个页面中重复初始化，而是有通用 `CommonChart`、复杂 `BubbleChart`、首页 `Map` 等组件。业务数据处理则通过 `useHomeOverview`、`useRegionData`、`useChartOption` 等 hooks 承担。

这种设计让页面更像“组合业务能力”，而不是“大段图表配置 + 请求逻辑 + DOM 操作”混在一起。

## 三、价值代码贡献评价

本节重点围绕 `FilterDropdowns.vue`、`BubbleChart.vue`、`Map.vue` 三个文件评价。评价项不逐条铺满所有细则，而是选取最能体现代码价值的三类要求：易扩展与契约化、代码整洁与可测试、高性能与复杂可视化处理。

| 要求 | 评价 |
| --- | --- |
| 易扩展 | `filterConfig` 能控制 Region/AZ 是否显示、资源类型使用 tree 还是 list、是否需要确认按钮 |

PPT 话术：  
`FilterDropdowns.vue` 最值得讲的是“一个组件支持多种筛选形态”。后续新增页面时，不需要重新写一套筛选 UI，只要调整 `filterConfig` 和传入的资源树结构，就能复用同一套交互和数据契约。

### 3. Map.vue：高性能与复杂可视化处理

`src/components/home/Map.vue` 负责世界地图、Region 散点、销毛率 visualMap、可拖拽标签和引导线。这个文件最适合体现高性能和复杂可视化处理：它把 GeoJSON 经度偏移、像素坐标换算、标签避让、ResizeObserver 资源回收等问题放在组件内部统一处理。

代表代码：

```js
const CUT_LONGITUDE = -30;

function shiftWorldGeoJSON(geojson) {
  const shifted = JSON.parse(JSON.stringify(geojson));
  shifted.features.forEach((feature) => {
    const { type, coordinates } = feature.geometry;
    if (type === "Polygon") {
      feature.geometry.coordinates = coordinates.map((ring) => processRing(ring));
    } else if (type === "MultiPolygon") {
      feature.geometry.coordinates = coordinates.map((polygon) =>
        polygon.map((ring) => processRing(ring))
      );
    }
  });
  return shifted;
}

function shiftLng(lng) {
  return lng < CUT_LONGITUDE ? lng + 360 : lng;
}
```
